# Recommendation Triage Plan

## 목적

이 문서는 현재 추천/상세 기능에서 발생하는 문제를 한 번에 정리하고,  
무엇이 실제 원인인지, 왜 그런 증상이 나오는지, 무엇부터 고쳐야 하는지 우선순위대로 정리하기 위한 문서다.

이 문서의 목표는 다음 3가지다.

1. 현재 문제를 증상 단위가 아니라 구조 단위로 이해한다.
2. 수정 우선순위를 명확히 정한다.
3. 구현 전에 팀이 같은 그림을 보도록 한다.

---

## 현재 관찰된 주요 증상

### 1. 코스 상세 조회에서 `코스를 찾을 수 없음`이 발생한다
- 메인 코스 또는 서브 코스 클릭 후 상세 조회가 실패한다.
- 특히 `other-courses`에서 내려준 항목을 클릭했을 때 실패할 가능성이 높다.

### 2. 메인 코스와 서브 코스가 너무 비슷하거나 거의 중복된다
- 같은 장소가 반복된다.
- 대표 이미지가 겹친다.
- 코스 구성이 main/sub1/sub2로 나뉘어도 체감상 거의 같은 코스처럼 보인다.

### 3. 장소 데이터 품질이 낮다
- 카페가 아닌 장소가 카페처럼 들어온다.
- 이색데이트/산책/음식점 분류가 어색하다.
- 지역은 맞아도 의도한 종류의 장소가 아닌 경우가 섞인다.

### 4. 이미지가 이상하거나 엉뚱하다
- 장소와 무관한 이미지가 붙는다.
- 건물 외관만 반복된다.
- 메인/서브에 같은 이미지가 반복된다.
- 카테고리와 안 맞는 이미지가 붙는다.

### 5. 추천 생성 API 응답이 느리다
- 장소 검색 이후 응답이 오래 걸린다.
- 이미지까지 붙는 경우 체감 지연이 더 커진다.
- 자동차 이동/경로 계산까지 들어가면 더 느려진다.

### 6. 스펙상 기대한 품질과 실제 구현 결과가 다르다
- 시간대 필터가 있어도 기대만큼 걸러지지 않는다.
- 이미지 필터링 규칙이 실제로는 거의 지켜지지 않는다.
- 분위기 일관성 규칙이 체감되지 않는다.

### 7. 지역과 시간대를 넣어도 `추천 결과를 찾을 수 없음`이 너무 자주 발생한다
- 사용자가 일반적인 지역과 시간대를 넣었는데도 추천 후보 부족 메시지가 자주 나온다.
- 일부 시간대에서는 추천이 거의 비어 보인다.
- late night뿐 아니라 낮/저녁 시간대에서도 기대보다 추천 실패가 많이 발생할 수 있다.

---

## 가장 중요한 결론

현재 문제는 단일 버그가 아니다.  
아래 4개의 구조 문제가 동시에 겹쳐서 발생하고 있다.

1. 저장/조회 구조 불일치
2. 외부 데이터 검증 부재
3. 다양성 제어 부재
4. 외부 API 호출 구조 비효율

즉, 지금 상태는 "조금만 고치면 끝나는 한두 개의 버그"가 아니라  
"추천 파이프라인 설계가 아직 정리되지 않은 상태"에 가깝다.

---

## 루트 원인 분석

## 1. 저장/조회 구조 불일치

### 현재 상태
- 추천 생성 결과는 서버 메모리 캐시에만 저장된다.
- 상세 조회는 DB를 기준으로 조회한다.
- `other-courses`는 실제 조회 가능한 서브 코스 ID가 아니라 가공된 ID를 내려준다.

### 왜 문제가 되는가
- 추천 생성 직후에는 메모리에만 존재하고 DB에는 없으므로 상세 조회가 실패할 수 있다.
- 서버가 재시작되면 메모리 데이터가 사라진다.
- 서버가 여러 인스턴스면 요청이 다른 인스턴스로 갔을 때 같은 코스를 찾지 못한다.
- 서브 코스는 실제 엔티티 ID가 없어서 상세 화면으로 자연스럽게 연결할 수 없다.

### 이 문제로 인해 발생하는 증상
- `코스를 찾을 수 없음`
- 서브 코스 클릭 실패
- 생성 응답과 상세 응답이 서로 다른 데이터 원천을 봄
- 프론트가 같은 코스를 다시 조합하려고 하며 불일치가 생김

### 결론
이 문제는 가장 먼저 정리해야 한다.  
저장하지 않을 거면 "다시 조회하는 API 구조"를 버려야 하고,  
다시 조회할 거면 "어딘가에는 저장하는 구조"가 반드시 필요하다.

---

## 2. 외부 데이터 검증 부재

### 현재 상태
- 네이버 검색 API 결과를 가져온 뒤 실제 업종/적합성을 충분히 검증하지 않는다.
- 검색어로만 후보를 가져오고, 결과가 정말 해당 카테고리에 맞는지 확인이 약하다.
- 실제 응답의 카테고리/설명/주소/품질을 검증하는 중간 정제 단계가 약하다.

### 왜 문제가 되는가
- `"성동구 카페"` 검색 결과에 카페가 아닌 장소가 섞여도 들어올 수 있다.
- 키워드만 비슷하면 산책/활동/음식점 분류가 어긋난다.
- 검색 결과 상위 노출이 곧 좋은 추천 후보라는 보장이 없다.

### 이 문제로 인해 발생하는 증상
- 네이버 데이터가 개판처럼 보임
- 장소 설명/분류가 어색함
- 코스 전체 일관성이 낮아짐

### 결론
`검색 -> 바로 코스 조합`이 아니라  
`검색 -> 정제 -> 점수화 -> 코스 조합` 구조가 필요하다.

---

## 3. 이미지 선정 검증 부재

### 현재 상태
- 장소별 이미지 검색 결과 중 첫 번째 "대충 괜찮아 보이는" 이미지를 채택하는 수준에 가깝다.
- 광고 키워드 일부를 제외하는 정도 외에 장소 일치도 검증이 약하다.
- 사람 얼굴 중심, 로고, 지도 캡처, 카테고리 부적합 이미지 등은 실제로 robust 하게 걸러지지 않는다.

### 왜 문제가 되는가
- 장소와 무관한 외관 사진이 쉽게 붙는다.
- 이미지 검색 결과 상위 노출 이미지가 반드시 좋은 카드 이미지가 아니다.
- 메인과 서브가 첫 장소를 공유하거나 유사하면 대표 이미지도 겹치기 쉽다.

### 이 문제로 인해 발생하는 증상
- 대표 이미지가 이상함
- 장소와 맞지 않는 사진
- 같은 건물 사진 반복
- 메인/서브 카드 이미지 중복

### 결론
이미지는 "있으면 붙인다"가 아니라  
"후보를 모아 점수화 후 가장 적절한 것을 선택한다"는 구조가 필요하다.

---

## 4. 코스 다양성 제어 부재

### 현재 상태
- 점수 상위 코스를 main/sub1/sub2로 단순 배정한다.
- 완전히 동일한 장소 집합만 중복으로 제거한다.
- 메인 코스와 많이 겹치는 서브 코스를 별도로 감점하거나 제외하지 않는다.

### 왜 문제가 되는가
- 장소 3개 중 2개가 겹쳐도 다른 코스로 남을 수 있다.
- 같은 첫 장소를 쓰면 대표 이미지도 동일해진다.
- 카테고리 순서와 분위기가 비슷한 코스가 그대로 서브로 채택된다.

### 이 문제로 인해 발생하는 증상
- 메인/서브 코스 중복
- 선택지가 여러 개인데 실제 체감은 거의 같음
- 서브 코스가 "대안"이 아니라 "복제본"처럼 보임

### 결론
서브 코스는 단순히 2등, 3등이 아니라  
"메인과 충분히 다른 후보" 중에서 뽑아야 한다.

---

## 5. 외부 API 호출 구조 비효율

### 현재 상태
- 장소 검색을 카테고리별로 순차 처리한다.
- 이미지 검색을 장소별로 순차 처리한다.
- 주차 검색을 장소별로 순차 처리한다.
- 경로 검색을 코스별 구간별로 순차 처리한다.
- 외부 HTTP 클라이언트를 호출마다 새로 생성한다.

### 왜 문제가 되는가
- 전체 응답 시간이 각 API 호출 시간의 합에 가까워진다.
- 장소 수가 많아질수록 이미지 검색이 병목이 된다.
- 자동차/경로 계산 포함 시 체감 지연이 크게 늘어난다.

### 이 문제로 인해 발생하는 증상
- 추천 생성 API가 느림
- 사용자가 화면에서 답답함을 느낌
- 외부 API 에러/타임아웃에 더 취약해짐

### 결론
병렬화, 연결 재사용, 캐시, 후처리 분리가 필요하다.

---

## 6. 스펙은 있지만 구현 가능한 판별 기준이 부족한 영역

### 문서 상 요구사항 중 구현 난이도가 높은 항목
- 사람 얼굴이 중심인 이미지 제외
- 카테고리와 맞지 않는 이미지 제외
- 영업 종료 1시간 이내 장소 제거
- 분위기 일관성 우선 선택

### 왜 문제가 되는가
- 네이버 검색 API만으로는 얼굴 중심 여부를 안정적으로 판별하기 어렵다.
- 이미지가 정말 장소/카테고리에 맞는지 텍스트만으로는 한계가 있다.
- 영업시간 데이터가 검색 결과에 안정적으로 제공되지 않을 수 있다.
- 분위기 일관성은 명확한 점수화 기준 없이는 구현이 흔들린다.

### 결론
이 항목들은 "규칙 존재"만으로는 부족하고  
"어떤 데이터 소스로 판별할지", "없으면 어떻게 degrade 할지"까지 정해야 한다.

---

## 7. 추천 후보 부족이 과하게 발생하는 구조

### 현재 상태
- 추천 코스는 기본적으로 `메인 1 + 서브 2`, 즉 장소 3개 조합을 전제로 한다.
- 카테고리 전환 순서가 비교적 엄격하다.
- 시간대 필터가 먼저 적용된다.
- 이동 가능 거리/시간 제약도 조합 과정에서 적용된다.
- 카테고리별로 상위 일부 후보만 사용해 조합한다.

### 왜 문제가 되는가
- 실제 외부 데이터 품질이 낮은 상태에서 규칙이 엄격하면 후보 풀이 급격히 줄어든다.
- 특정 지역은 카테고리별 후보 수가 고르지 않을 수 있다.
- 시간대 필터 이후 특정 카테고리가 거의 비면 조합 자체가 성립하지 않는다.
- 거리 제한까지 걸리면 "후보는 있는데 조합이 안 되는" 상황이 자주 생긴다.
- late night처럼 허용 조건이 좁은 슬롯은 특히 실패율이 높아진다.

### 후보 부족이 자주 생기는 구체적 이유

#### 1. 카테고리별 후보 수 불균형
- 예를 들어 카페는 많은데 activity/walk가 거의 없으면 3개 조합이 어렵다.
- 지역별 특성상 특정 카테고리가 약한 경우가 있다.

#### 2. 시간대 필터가 적용된 뒤 후보 풀이 급감
- morning에는 음식점 제외
- late_night에는 카페/음식점/활동 허용 조건이 강함
- 이 상태에서 실제 키워드/영업시간 정보가 부족하면 사실상 많이 탈락하거나 반대로 유효 후보 판단이 흔들린다.

#### 3. 조합 규칙이 엄격한데 fallback이 약함
- 시작 카테고리와 전환 규칙을 따르다 보면 특정 순서 조합만 시도하게 된다.
- 한 카테고리가 비면 대체 전략이 충분하지 않다.
- "3개 코스가 안 되면 2개 코스라도 제안" 같은 단계적 완화 전략이 없다.

#### 4. 이동 거리/시간 제한으로 조합이 더 줄어든다
- 장소 자체는 있어도 서로 멀면 코스 후보에서 탈락한다.
- 지역명이 같아도 실제 좌표가 넓게 퍼져 있으면 조합 실패가 많아진다.

#### 5. 검색 결과 상위 일부만 써서 풀 자체가 좁다
- 각 카테고리별 상위 일부 후보만 사용하면 조합 다양성이 줄어든다.
- 상위 노출 장소가 서로 멀거나 품질이 낮으면 아래 후보를 보지 못하고 실패한다.

### 이 문제로 인해 발생하는 증상
- 사용자는 "지역/시간 넣었는데 왜 아무 것도 안 나오지?"라고 느낀다.
- 실제로는 장소가 없는 게 아니라, 현재 규칙으로는 조합이 안 되는 상태일 수 있다.
- 추천 시스템 신뢰도가 크게 떨어진다.

### 결론
`추천 결과 없음`이 자주 나오는 것은 단순히 데이터가 없어서가 아니라  
`후보 수집 품질 + 조합 규칙 강도 + fallback 부재`가 같이 만든 문제다.

---

## 이 문서 기준 최우선 판단

가장 먼저 해결해야 하는 것은 `이미지 품질`도 아니고 `속도`도 아니다.  
가장 먼저 해결해야 하는 것은 `저장/조회 구조`다.

이유는 다음과 같다.

1. 조회가 안 되면 기능 자체가 깨진다.
2. 데이터 구조가 정리되지 않으면 나머지 개선이 누적되지 않는다.
3. 서브 코스 ID 체계가 정리되지 않으면 프론트 연결도 계속 불안정하다.
4. 캐시/DB/TTL 전략이 정해져야 이후 API 응답 구조도 안정화된다.

## 무엇부터 해야 하는가

## 1순위: 저장/조회 아키텍처 결정

### 반드시 먼저 결정할 것
- 추천 결과를 정말 저장하지 않을 것인지
- 다시 조회 가능한 상세 API를 유지할 것인지
- 메인/서브 각각을 실제 개별 코스로 식별할 것인지

### 권장 방향
영구 DB 저장 대신 `TTL 기반 저장소`를 사용한다.

추천:
- MySQL: 이벤트, 영구 로그, 분석용 데이터
- Redis 또는 TTL 캐시 저장소: 추천 결과

### 이유
- 추천 결과는 임시 데이터 성격이 강하다.
- 그러나 상세 조회/서브 코스 클릭/다시 불러오기를 위해서는 재조회 가능한 저장소가 필요하다.
- TTL 캐시 저장소는 DB 과적재 걱정을 줄이면서 조회 가능성을 보장한다.

### 이 단계 완료 조건
- `recommendation_id`와 `course_id` 구조 확정
- main/sub1/sub2 모두 실제 조회 가능한 ID 부여
- 상세 API와 추천 API가 같은 저장소를 보도록 통일
- TTL 만료 정책 정의

---

## 2순위: 응답/식별자 모델 재설계

### 반드시 정리할 것
- 추천 세트 식별자와 개별 코스 식별자를 분리
- `other-courses`에 실제 조회 가능한 `course_id` 제공
- 생성 응답에서 프론트가 후속 조회에 필요한 ID를 모두 받을 수 있게 설계

### 권장 구조
- `recommendation_id`: 추천 요청 1회 전체 묶음
- `course_id`: main, sub1, sub2 각각의 상세 조회용 ID

### 이 단계 완료 조건
- 가짜 식별자 제거
- 프론트가 `course_id` 기반으로 안정적으로 상세 조회 가능

## 3순위: 장소 후보 정제 레이어 추가

### 목표
네이버 검색 결과를 곧바로 추천 후보로 쓰지 않도록 한다.

### 최소 필요 규칙
- 지역 일치 여부 검증
- 좌표 유효성 검증
- 실제 카테고리 적합성 검증
- 광고성/노이즈 결과 제외
- 중복 장소 제거
- 설명/메타데이터가 너무 빈약한 결과 후순위 처리

### 이 단계 완료 후 기대 효과
- 데이터 품질 개선
- 이상한 장소 감소
- 코스 일관성 향상

## 4순위: 코스 다양성 규칙 추가

### 목표
main/sub1/sub2가 서로 충분히 다른 대안이 되도록 한다.

### 추가해야 할 규칙
- 메인과 장소 2개 이상 겹치는 서브 후보 제외 또는 강한 감점
- 첫 장소 동일 시 감점
- 카테고리 순서 동일 시 감점
- 대표 이미지 동일 시 감점
- 분위기 키워드가 거의 동일하면 감점

### 이 단계 완료 후 기대 효과
- 메인/서브 체감 다양성 증가
- 추천 결과 만족도 개선

## 5순위: 추천 실패율 완화 전략 추가

### 목표
`추천 결과 없음`을 가능한 한 줄이고, 정말 불가한 경우에만 부족 메시지를 반환한다.

### 추가해야 할 전략
- 카테고리 후보 부족 시 완화 가능한 fallback 순서 정의
- 3개 조합이 불가능하면 2개 조합 허용 여부 결정
- 카테고리 전환 규칙 실패 시 대체 순서 허용
- 상위 후보만 보지 말고 조합 실패 시 후보 풀을 단계적으로 확장
- 거리 제한/품질 제한을 한 번에 모두 강하게 적용하지 말고 단계적 완화 전략 도입

### 권장 방향
- 1차: 스펙 기준 엄격 추천
- 2차: 후보 풀 확장
- 3차: 일부 규칙 완화
- 4차: 정말 불가능할 때만 부족 메시지 반환

### 이 단계 완료 후 기대 효과
- 추천 생성 성공률 증가
- 사용자 체감 "추천 안 나옴" 감소
- 지역별/시간대별 실패 편차 완화

## 6순위: 이미지 선정 로직 개선

### 목표
이미지가 "아무거나 붙는 상태"를 벗어나게 한다.

### 추가해야 할 규칙

- 카테고리별 허용 이미지 유형 점수
- 광고/배너/협찬/부동산/지도/로고 제거 강화
- 동일 이미지 반복 방지
- 이미지 없을 때는 과감히 빈값 허용

### 주의
이미지 품질을 너무 엄격하게 잡으면 응답 속도가 악화될 수 있다.  
필요하면 이미지 보강은 후처리로 분리한다.

## 지금 당장 하지 말아야 할 것

### 1. 이미지 품질만 먼저 손보는 것
- 저장/조회 구조가 깨져 있으면 효과가 누적되지 않는다.

### 2. 프론트에서 임시로 우회 로직을 많이 넣는 것
- 현재 문제는 백엔드 구조 문제 비중이 크다.
- 프론트 임시조치는 불일치만 키울 수 있다.

### 3. 추천 결과를 영구 DB에 무작정 쌓는 것
- 운영 데이터만 비대해질 수 있다.
- 저장이 필요하다면 TTL 전략부터 정해야 한다.

### 4. 스펙 문구만 믿고 판별 가능한 데이터 없이 구현을 밀어붙이는 것
- 얼굴 중심 이미지 제외, 분위기 일관성 같은 규칙은 판별 방법이 먼저다.

## 추천 아키텍처 방향

## 권장안: TTL 저장소 기반 추천 결과 관리

### 개요
- 추천 생성 결과는 `Redis` 같은 TTL 저장소에 저장한다.
- main/sub1/sub2 각각 실제 `course_id`를 가진다.
- 상세 조회는 같은 TTL 저장소를 본다.
- TTL 만료 후에는 "추천이 만료되었습니다. 다시 생성해 주세요"로 처리한다.

### 장점
- DB 용량 부담이 적다.
- 상세 조회와 서브 코스 연결이 가능하다.
- 단일 서버 메모리보다 안정적이다.
- 멀티 인스턴스 환경에서도 일관성이 생긴다.

### 단점
- Redis 또는 유사 저장소가 필요하다.
- TTL/만료 UX를 설계해야 한다.

### 왜 현재 상황에 맞는가
- 추천 결과는 영구 보존 가치보다 단기 조회 가치가 크다.
- 사용자는 생성 직후 상세 보기/서브 코스 비교가 더 중요하다.

## 팀 의사결정이 필요한 항목

다음 항목은 구현 전에 반드시 결정해야 한다.

### 1. 추천 결과를 얼마 동안 보관할 것인가
- 예: 30분 / 1시간 / 24시간

### 2. TTL 만료 후 UX는 어떻게 할 것인가
- 재추천 유도
- 이전 검색 조건으로 다시 생성

### 3. 이미지 품질과 응답 속도 중 무엇을 우선할 것인가
- 응답 우선이면 이미지 후처리
- 품질 우선이면 동기 처리 + 병렬화

### 4. 영업시간/분위기 정보가 부족할 때 어떻게 degrade 할 것인가
- 정보 부족이면 통과
- 정보 부족이면 감점
- 특정 카테고리만 엄격 적용

### 5. 추천 후보가 부족할 때 어디까지 완화할 것인가
- 후보 풀 확장만 할지
- 조합 규칙도 완화할지
- 2개 코스도 허용할지
- 부족 메시지를 언제 반환할지

## 최종 정리

현재 가장 먼저 해야 할 일은 다음 3가지다.

1. 추천 결과 저장/조회 구조를 통일한다.
2. 메인/서브 코스 식별자 체계를 다시 설계한다.
3. 장소 후보 정제 레이어를 넣는다.
4. 추천 실패율을 낮추는 fallback 전략을 넣는다.

이 3가지가 해결되기 전에는:
- 이미지 품질을 올려도 불안정하고
- 성능을 최적화해도 품질이 낮고
- 프론트에서 우회해도 구조 문제가 남는다.

따라서 작업 순서는 반드시 아래처럼 가져가는 것이 좋다.

1. 구조 안정화
2. 응답 모델 정리
3. 후보 정제
4. 추천 실패율 완화
5. 다양성 강화
6. 이미지 개선
7. 성능 최적화

이 문서는 "지금 무엇이 문제인가"에 대한 공통 기준이며,  
이후 실제 작업은 이 우선순위를 기준으로 진행하는 것을 권장한다.
